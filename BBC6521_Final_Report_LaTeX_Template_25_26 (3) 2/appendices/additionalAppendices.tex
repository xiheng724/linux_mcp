\subsubsection*{A.1\quad Wire Protocol Reference}
\phantomsection
\addcontentsline{toc}{subsubsection}{A.1\quad Wire Protocol Reference}
\label{sec:appendix-protocols}

This section provides the complete field specifications and representative implementation fragments for the three inter-component protocols described in Section~\ref{sec:wire-protocols}.

\paragraph{Complete llm-app to mcpd message reference.}

The \texttt{tool:exec} request carries the tool hash as a 64-character hexadecimal string.
If a prior \texttt{DEFER} decision returned a \texttt{ticket\_id}, the client includes it in a replay request:

\begin{lstlisting}[style=codestyle, language=json, caption={tool:exec request and response messages (llm-app $\to$ mcpd).}]
// tool:exec request
{ "kind": "tool:exec",  "req_id": 42,
  "session_id": "sess-abc", "agent_id": "agent-1",
  "app_id": "sys_app",  "tool_id": 1,
  "tool_hash": "a3f2...c91b",        // 64-char SHA-256 hex
  "payload": {"path": "/tmp"},
  "approval_ticket_id": null }       // set on replay after DEFER

// tool:exec response (ALLOW)
{ "req_id": 42, "status": "ok",
  "result": {"os": "Linux 6.8", "cpu": "4-core"},
  "decision": "ALLOW", "reason": "admitted",
  "ticket_id": 0,  "t_ms": 3 }

// tool:exec response (DEFER)
{ "req_id": 42, "status": "error",
  "result": {}, "error": "deferred pending approval",
  "decision": "DEFER", "reason": "risk_tag:sensitive_read",
  "ticket_id": 7,  "t_ms": 1 }
\end{lstlisting}

The \texttt{approval\_reply} message is issued by the client after the user confirms a deferred operation:

\begin{lstlisting}[style=codestyle, language=json, caption={approval\_reply message (llm-app $\to$ mcpd, issued after user confirms a deferred operation).}]
// approval_reply request
{ "sys": "approval_reply",  "session_id": "sess-abc",
  "ticket_id": 7,  "decision": "approve",
  "reason": "user confirmed",  "ttl_ms": 30000 }
\end{lstlisting}

\paragraph{mcpd to tool-app message schema.}

Once the kernel returns \texttt{ALLOW}, \texttt{mcpd} forwards the request to the relevant tool service over a per-application UDS connection using the minimal five-field schema below.
Tool services implement only this contract; they require no knowledge of sessions, kernel tickets, or the Netlink protocol.
The \texttt{operation} field is the dispatch string a tool service routes on and is declared per-tool in the manifest; it is distinct from the \texttt{tool\_id} (numeric, used by the kernel for registry indexing) and from the tool's human-readable \texttt{name}.

\paragraph{Complete Netlink attribute encoding.}

All attributes are 4-byte aligned.
String attributes are NUL-terminated and passed as raw bytes.
The following Python fragment shows how \texttt{mcpd} encodes a \texttt{TOOL\_REQUEST} message:

\begin{lstlisting}[style=codestyle, language=Python, caption={Python fragment: encoding a \texttt{TOOL\_REQUEST} Generic Netlink message (mcpd $\to$ kernel).}]
import struct, socket, hashlib

FAMILY_NAME        = b"KERNEL_MCP\x00"
CMD_TOOL_REQUEST   = 10
ATTR_AGENT_ID,   ATTR_TOOL_ID     = 4,  2
ATTR_REQ_ID,     ATTR_TOOL_HASH   = 1,  19
ATTR_AGENT_BINDING, ATTR_AGENT_EPOCH = 28, 29

def build_nla(attr_type, data):
    # NLA header: 2-byte length, 2-byte type; 4-byte aligned
    hdr = struct.pack("HH", len(data) + 4, attr_type)
    pad = (4 - len(data) % 4) % 4
    return hdr + data + b"\x00" * pad

def encode_tool_request(agent_id, tool_id, req_id,
                        tool_hash, binding, epoch):
    attrs = (
        build_nla(ATTR_AGENT_ID,       agent_id.encode() + b"\x00") +
        build_nla(ATTR_TOOL_ID,        struct.pack("=I", tool_id))   +
        build_nla(ATTR_REQ_ID,         struct.pack("=Q", req_id))    +
        build_nla(ATTR_TOOL_HASH,      tool_hash.encode() + b"\x00") +
        build_nla(ATTR_AGENT_BINDING,  struct.pack("=Q", binding))   +
        build_nla(ATTR_AGENT_EPOCH,    struct.pack("=Q", epoch))
    )
    # Generic Netlink header: cmd (1 B), version (1 B), reserved (2 B)
    genl_hdr = struct.pack("BBH", CMD_TOOL_REQUEST, 1, 0)
    return genl_hdr + attrs
\end{lstlisting}

\paragraph{Semantic hash computation.}

The semantic hash used for tool identity is a full 256-bit SHA-256 digest computed over a canonical JSON serialisation of the following manifest fields: \texttt{tool\_id}, \texttt{name}, \texttt{app\_id}, \texttt{app\_name}, \texttt{risk\_tags}, \texttt{description}, \texttt{input\_schema}, \texttt{examples}, \texttt{path\_semantics}, \texttt{approval\_policy}, and \texttt{operation}.
The \texttt{endpoint} and \texttt{transport} fields are deliberately excluded so that redeployment to a new socket path or transport does not invalidate the kernel registration; \texttt{operation} remains in-hash because it selects the RPC method actually executed on the backend.

\begin{lstlisting}[style=codestyle, language=Python, caption={Python fragment: computing the manifest-derived semantic hash used for tool identity.}]
import json, hashlib

SEMANTIC_FIELDS = [
    "tool_id", "name", "app_id", "app_name",
    "risk_tags", "description", "input_schema",
    "examples", "path_semantics", "approval_policy",
]

def semantic_hash(tool_manifest: dict) -> str:
    canonical = {f: tool_manifest[f] for f in SEMANTIC_FIELDS}
    blob = json.dumps(canonical,
                      sort_keys=True,
                      separators=(",", ":"),
                      ensure_ascii=True).encode("utf-8")
    return hashlib.sha256(blob).hexdigest()  # 64-char hex; never truncated
\end{lstlisting}

The full 64-character digest is used without truncation.
A 32-bit prefix (8 hex characters) would be vulnerable to birthday collisions after approximately $2^{16}$ distinct tool registrations, which is a realistic size for a production deployment spanning multiple applications.

\subsubsection*{A.3\quad Experiment Kernel Interface}
\phantomsection
\addcontentsline{toc}{subsubsection}{A.3\quad Experiment Kernel Interface}
\label{sec:appendix-experiment-uapi}

The latency, stage-ablation, and extended-attack experiments in \hyperref[chap:evaluation]{Chapter~4} rely on three kernel-side additions that live behind compile-time and module-parameter switches so that a single binary can serve both production and benchmark roles.
This section documents their UAPI footprint and the peer-credential patch surfaced by the cross-UID attack rerun.

\paragraph{Noise-floor command: \texttt{KMCP\_CMD\_NOOP}.}

The paired ablation anchors every stage cost against a Generic Netlink round-trip floor measured by a minimum-work handler that returns immediately after command dispatch.
The command is appended to the existing \texttt{KERNEL\_MCP} family at a stable numeric ID so that older \texttt{mcpd} builds continue to work unchanged:

\begin{lstlisting}[style=codestyle, language=C, caption={UAPI additions for the noise-floor command and the stage-ablation experiment flags (kernel-mcp/include/uapi/linux/kernel\_mcp\_schema.h).}]
/* New command: stable numeric ID, append-only. */
#define KERNEL_MCP_CMD_NOOP                15

/* Compile-time experiment flags carried in KERNEL_MCP_ATTR_EXPERIMENT_FLAGS
 * on KMCP_CMD_TOOL_REQUEST. Unknown bits are ignored. */
#define KERNEL_MCP_EXPERIMENT_SKIP_LOOKUPS  (1U << 0)
#define KERNEL_MCP_EXPERIMENT_SKIP_HASH     (1U << 1)
#define KERNEL_MCP_EXPERIMENT_SKIP_BINDING  (1U << 2)
#define KERNEL_MCP_EXPERIMENT_SKIP_TICKET   (1U << 3)
\end{lstlisting}

The \texttt{SKIP\_*} flags are honoured only when the kernel is built with \texttt{CONFIG\_KERNEL\_MCP\_EXPERIMENT=y}; production builds ignore the attribute entirely, so the mere presence of the UAPI constants does not weaken the enforcement path.
Under the benign workload used by the ablation runner, the \texttt{SKIP\_HASH} and \texttt{SKIP\_BINDING} guards are statically dead code (short-circuited by a valid request before the flag is consulted) and their paired deltas correctly return zero to within noise; they are retained for methodology-consistency checks rather than as meaningful stage costs.

\paragraph{Peer-credential enforcement: \texttt{require\_peer\_cred}.}

The cross-UID attack rerun surfaced a defect in the stock kernel module: \texttt{kernel\_mcp\_cmd\_tool\_request} never compared the \texttt{NETLINK\_CB(skb)} peer credential against the \texttt{uid} recorded at agent registration, which allowed a crafted-credential replay from a second UID to reach the tool-request handler untouched.
The fix is exposed as a module parameter so that the A/B comparison in \hyperref[chap:evaluation]{Chapter~4} could run against a single binary:

\begin{lstlisting}[style=codestyle, language=C, caption={Peer-credential enforcement patch in \texttt{kernel\_mcp\_main.c}, gated behind the \texttt{require\_peer\_cred} module parameter.}]
static bool require_peer_cred = false;
module_param_named(require_peer_cred, require_peer_cred, bool, 0644);
MODULE_PARM_DESC(require_peer_cred,
    "Reject TOOL_REQUEST if NETLINK_CB peer UID != agent->uid");

static int kernel_mcp_cmd_tool_request(struct sk_buff *skb,
                                       struct genl_info *info)
{
    if (require_peer_cred) {
        kuid_t peer = sock_i_uid(NETLINK_CB(skb).sk);
        if (!uid_eq(peer, agent->uid))
            return -EPERM;           /* cross-uid replay; reject */
    }
    /* ... existing admission pipeline ... */
}
\end{lstlisting}

The parameter defaults to \texttt{false} in the artefact published alongside this report so that the extended-attack experiment can exercise both the pre-patch and post-patch code paths without a rebuild; production deployments should set it to \texttt{true} via \texttt{/sys/module/kernel\_mcp/parameters/require\_peer\_cred} on module load.
The check reads the peer credential on the same cache line that the existing \texttt{NETLINK\_CB} access already touches, so the \hyperref[chap:evaluation]{Chapter~4} latency and throughput numbers are unchanged whether the gate is on or off.
