\subsection{Conclusion}

The broader question behind this project is what a traditional Linux system needs in order to host AI agents well.
This work takes the narrower angle of asking it through the MCP protocol: if an agent's interaction with the outside world flows through a standardised tool-invocation layer, which parts of that layer genuinely need operating-system support, and which are fine where they are?

For the control-plane side of MCP, the answer is a small set of invariants: semantic consistency between a registered tool and the one actually executed, identity binding between a request and the session that made it, approval validity across the defer-approve-execute lifecycle, and state durability across gateway failure.
These four properties are structurally hard to hold in a user-space gateway, because each one has to survive the failure or compromise of the very process that enforces it.
Everything else belongs in user space, where flexibility matters and mistakes are cheap to fix.

\texttt{linux-mcp} is one concrete shape this split can take.
A small Linux kernel module holds the registries, arbitrates admission, and owns the approval-ticket lifecycle behind a \texttt{CAP\_\allowbreak NET\_\allowbreak ADMIN}-gated Netlink interface; a user-space daemon loads manifests, derives semantic hashes, binds sessions to UDS peer credentials, and forwards admitted requests to tool services.
Once the daemon has populated the kernel's view of the world, the integrity of that view no longer depends on the daemon staying alive or correct, and the tickets and registry entries stay in kernel memory after a daemon crash.

In the tested prototype, the kernel path enforced all specified control-plane invariants across the targeted attack set while both user-space baselines permitted meaningful bypass rates; latency and throughput stayed close to the user-space baseline throughout.
A separate extended-attack rerun (TOCTOU, cross-UID session hijack, 30-minute Generic Netlink dumb fuzzer) surfaced a missing peer-credential check in the stock kernel module and closed it with a six-line patch at no measurable latency cost.
These results are a functional-correctness test of specified invariants inside a single VM, not a claim of adversarial robustness or production-scale performance.
They are nevertheless enough to show the split is workable: the enforcement gains are real, the overhead is small, and neither the LLM client nor tool authors need to change anything beyond providing a JSON manifest.

The position argued here is narrower than AIOS-style proposals that rebuild OS infrastructure around LLM abstractions, and less trusting than user-space frameworks that accept gateway correctness as given.
Whether AI agents eventually deserve a full operating system of their own is an open question and probably a long way off; in the meantime, a conventional Linux system extended with a few carefully scoped agent-aware mechanisms can already host an MCP control plane that neither a user-space gateway nor existing Linux security primitives can match alone.
\texttt{linux-mcp} is one such mechanism for one such concern, and finding the others is the longer-term project.

\subsection{Reflection}

The initial framing of this project was too broad.
Work became substantially clearer once the problem was narrowed to a specific question: which invariants break when the gateway crashes or misbehaves, and are those the ones that benefit from kernel enforcement?
That reframing aligned the threat model, the implementation, and the evaluation criteria.

The kernel interface went through several candidates before settling on Generic Netlink.
eBPF was considered first, and state storage turned out not to be the obstacle; \texttt{bpf\_\allowbreak timer} and pinned maps are adequate for approval-ticket lifecycle.
The real obstacle is the hook-attachment model: eBPF LSM fires on kernel operations, not on parsed JSON-RPC messages, so an in-kernel enforcer would have to re-parse MCP framing from raw socket bytes under verifier bounded-loop constraints on every invocation.
LSM hooks were also considered, but they fire at the syscall boundary while tool identity is an application-layer concept, and the layers do not align.
Generic Netlink supports the full lifecycle state the design needs, at the cost of a loadable module where a simpler policy hook would have been lighter.

Working across the kernel-user-space boundary was technically demanding.
Attribute packing bugs in Netlink produce no compiler errors; they silently corrupt fields, surfacing only as unexpected kernel decisions deep into a test run.
The most useful mitigation was writing standalone per-command test scripts before integrating with \texttt{mcpd}.

The most persistent design tension was the temptation to move more logic into the kernel.
Each additional check there is one the gateway cannot bypass, which is attractive, but manifest parsing, schema validation, and tool execution belong in user space where iteration is fast and debugging does not require a module reload.
The narrow kernel role was not a limitation; it was the point.

More broadly, the scope of a security claim matters as much as the claim itself.
Framing \texttt{linux-mcp} as protecting four specific control-plane properties against a local unprivileged adversary made the evaluation honest and the argument defensible.
A system that claims to solve the full AI-agent security problem is harder to evaluate and trust than one that solves a well-defined sub-problem and says so.

\subsection{Further Work}
\label{sec:further-work}

Several directions follow naturally from the prototype.

First, kernel arbitration is intentionally minimal: tool and agent identity, semantic-hash consistency, and approval-ticket validity, but no richer policy framework.
Per-tool permissions, role-scoped access control, rate limits, and execution budgets are the obvious next step, subject to the narrow-kernel principle that the kernel verifies policy membership but never computes policy decisions.

Second, the approval mechanism can be developed into a fuller governance path.
The present design supports defer-and-decide control for sensitive operations, but real deployments will likely need approval delegation, bounded approval scope, revocation, expiry policies, and richer audit trails linking each decision to a concrete execution context.
Strengthening this path would make \texttt{linux-mcp} more suitable for accountability and compliance contexts.

Third, recovery behaviour can be refined.
The prototype preserves kernel-held registry and approval state across daemon failure, but cold-start reconciliation is unconditional, and daemon-local state (sessions and the \texttt{req\_\allowbreak id}-to-ticket map) is discarded rather than reconstructed, so clients must reopen a session before resuming.
A more mature design would gate the cold-start sweep on an observable divergence signal, carry richer recovery metadata for interrupted requests, and let clients resume a still-valid approval ticket without reopening.
This matters most in long-running agent workflows, where failure and restart are routine.

Fourth, the evaluation can be extended substantially.
The current registry-scaling experiment already pushes the kernel catalogue to \num{16384} tools with flat lookup latency across three orders of magnitude of $N$, but several adjacent regimes remain uncharacterised: bare-metal evaluation to eliminate hypervisor timing noise, sustained concurrency well beyond the clean-sweep ceiling of 100 concurrent requests (the \texttt{SOMAXCONN} saturation that blocked the present overload attempt needs addressing in the harness first), heterogeneous tool workloads with variable or network-dependent execution times, and $p_{99}$ latency under sustained concurrency (the $p_{95}$ characterisation in \hyperref[chap:evaluation]{Chapter~4} does not extend that far).
Comparison against additional hardened user-space designs would help separate the benefits of kernel placement from those achievable through improved user-space engineering alone.

Finally, this work opens a broader research question: which other agent-oriented abstractions deserve operating-system support?
\texttt{linux-mcp} shows that a small but consequential part of the MCP control plane benefits from OS-level support without requiring AI agents to become first-class OS citizens or a new OS substrate.
Similar arguments may extend to execution budgets, provenance across tool chains, durable approval logs, or agent-scoped resource accounting.
Exploring these directions would clarify whether agent-aware OS support is best added incrementally, as in the present design, or eventually justifies a dedicated substrate.
