\subsection{Conclusion}

The broader question behind this project is what a traditional Linux system actually needs in order to host AI agents well.
This work takes the narrower angle of asking it through the MCP protocol: if an agent's interaction with the outside world flows through a standardised tool-invocation layer, which parts of that layer genuinely need operating-system support, and which are fine where they are?

The answer, at least for the control-plane side of MCP, is a small set of invariants: semantic consistency between a registered tool and the one actually executed, identity binding between a request and the session that made it, approval validity across the defer--approve--execute lifecycle, and state durability across gateway failure.
These four properties are structurally hard to hold in a user-space gateway, because every one of them has to survive the failure or compromise of the very process that enforces them.
Everything else has no such requirement and belongs in user space, where flexibility matters and mistakes are cheap to fix.

\texttt{linux-mcp} is one concrete shape this split can take.
A small Linux kernel module holds the registries, arbitrates admission, and owns the approval-ticket lifecycle behind a \texttt{CAP\_NET\_ADMIN}-gated Netlink interface; a user-space daemon loads manifests, derives semantic hashes, binds sessions to UDS peer credentials, and forwards admitted requests to tool services.
The gateway is still fully trusted to populate the kernel's view of the world, but once that view is populated, its integrity no longer depends on the daemon staying alive or staying correct.
Approval tickets and registry entries sit in kernel memory, out of reach of unprivileged processes, and still there after a daemon crash.

In the tested prototype, the kernel path enforced all specified control-plane invariants across the targeted attack set, while both user-space baselines permitted meaningful bypass rates; latency and throughput remained close to the user-space baseline throughout.
A separate extended-attack rerun covering TOCTOU, cross-UID session hijack, and a 30-minute Generic Netlink dumb fuzzer additionally surfaced a missing peer-credential check in the stock kernel module — the kind of absence that design-time review is unlikely to catch — and closed it with a six-line patch at no measurable latency cost.
These results are not intended as a claim of adversarial robustness or production-scale performance, since the attack set remains a functional-correctness test of the specified invariants and the evaluation was conducted inside a single VM.
They are nevertheless sufficient to show that the split is workable: the enforcement gains are real, the overhead is small, and neither the LLM client nor tool authors need to change anything beyond providing a JSON manifest.

The position this work ends up arguing for is narrower than AIOS-style proposals that rebuild OS infrastructure around LLM abstractions, and less trusting than user-space frameworks that accept gateway correctness as a given.
Whether AI agents eventually deserve a full operating system of their own is an open question and probably a long way off.
In the meantime, a useful intermediate answer is that a conventional Linux system, extended with a small number of carefully scoped agent-aware mechanisms, can already host an MCP control plane that neither a user-space gateway nor existing Linux security primitives can match on its own.
\texttt{linux-mcp} is one such mechanism, for one such concern.
Finding the others is the actual long-term project.

\subsection{Reflection}

The initial framing of this project was too broad; the work became substantially clearer once the problem was narrowed to a specific question: which invariants break when the gateway crashes or misbehaves, and are those the ones that benefit from kernel enforcement?
That reframing aligned the threat model, the implementation, and the evaluation criteria.

On the technical side, the kernel interface went through several candidates before settling on Generic Netlink.
eBPF was the first choice, but the approval-ticket lifecycle requires stateful tracking across multiple messages that the verifier's loop and memory restrictions make impractical.
LSM hooks were also considered, but they operate at the syscall boundary while tool identity is an application-layer concept; the layers do not align.
Generic Netlink supports the full lifecycle state the design needs, though it requires a loadable module where a simple policy hook would have been lighter.

Working across the kernel--user-space boundary was technically demanding.
Attribute packing bugs in Netlink do not produce compiler errors; they silently corrupt fields, with failures only appearing as unexpected kernel decisions deep into a test run.
The most useful mitigation was building standalone per-command test scripts before integrating with \texttt{mcpd}, which made failures easier to isolate.

The most persistent design tension was the temptation to move more logic into the kernel.
Each additional check there is one the gateway cannot bypass, which is attractive.
But manifest parsing, schema validation, and tool execution belong in user space, where iteration is fast and debugging does not require a module reload.
The narrow kernel role was not a limitation of the design; it was the point.

More broadly, the scope of a security claim matters as much as the claim itself.
Framing \texttt{linux-mcp} as protecting four specific control-plane properties against a local unprivileged adversary made the evaluation honest and the argument defensible.
A system that claims to solve the full AI-agent security problem is harder to evaluate and harder to trust than one that solves a well-defined sub-problem and says so clearly.

\subsection{Further Work}
\label{sec:further-work}

Several directions follow naturally from the current prototype.

First, the current kernel arbitration logic is intentionally minimal.
It verifies tool identity, agent identity, semantic-hash consistency, and approval-ticket validity, but it does not implement a richer policy framework.
The next step is to support more expressive admission policies, such as per-tool permissions, role-scoped access control, rate limits, or execution budgets.
The design principle should be maintained: the kernel verifies policy membership but does not compute policy decisions, which remain in user space, preserving the narrow-kernel invariant.

Second, the approval mechanism can be developed into a more complete governance path. 
The present design supports defer-and-decide control for sensitive operations, but real deployments may require approval delegation, bounded approval scope, revocation, expiry policies, and richer audit trails linking each decision to a concrete execution context. 
Strengthening this path would make \texttt{linux-mcp} more suitable for environments with accountability or compliance requirements.

Third, recovery behaviour can be improved further. The current prototype already preserves kernel-held registry and approval state across daemon failure, but reconciliation is deliberately conservative and session-level state is rebuilt after restart. 
A more mature design could provide more systematic restart handling, richer recovery metadata, and clearer support for interrupted or partially completed requests.
This becomes particularly important for long-running agent workflows, where failure and restart are routine occurrences.

Fourth, the evaluation can be extended substantially.
The current study includes a dedicated registry-scaling experiment that already pushes the kernel catalogue to \num{16384} tools and shows flat lookup latency across three orders of magnitude of $N$, but several adjacent regimes remain uncharacterised.
Future work should include bare-metal evaluation to eliminate hypervisor timing noise, sustained concurrency levels well beyond the current clean sweep ceiling of 100 concurrent requests (the \texttt{SOMAXCONN} saturation that blocked the present overload attempt needs addressing at the benchmark harness first), and heterogeneous tool workloads with variable or network-dependent execution times.
$p_{95}$ latency is characterised in \hyperref[chap:evaluation]{Chapter~4}; $p_{99}$ under sustained concurrency remains unmeasured and is a natural next step.
Comparison against additional hardened user-space designs would also help sharpen the argument by separating the benefits of kernel placement from those achievable through improved user-space engineering alone.

Finally, this work opens a broader research question: which other agent-oriented abstractions deserve operating-system support?
\texttt{linux-mcp} shows that a small but consequential part of the MCP control plane benefits from OS-level support without requiring AI agents to become first-class OS citizens or requiring a new OS substrate.
This project focuses on admission control for tool invocation, but similar arguments may extend to execution budgets, provenance across tool chains, durable approval logs, or agent-scoped resource accounting.
Exploring these directions would help clarify whether support for AI agents should be understood as entirely new operating system substrates.
