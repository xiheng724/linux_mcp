\phantomsection \label{chap:design} \setcounter{section}{3} \setcounter{subsection}{0}
\addcontentsline{toc}{section}{\textit{Chapter 3: Design and Implementation}}

The design question for linux-mcp is a question of placement: which parts of the control plane should the gateway not be solely responsible for holding?
\texttt{mcpd} keeps everything that needs semantic flexibility; \texttt{kernel\_\allowbreak mcp} owns the state that must survive gateway failure and stay unreachable from unprivileged processes.



\subsection{Design Goals}
\label{sec:design-goals}

Chapter~\ref{chap:background} established that MCP-style control planes need enforcement beyond user-space self-policing. 
linux-mcp translates that conclusion into three design commitments.

\textbf{(1) Kernel-resident admission.}
Every tool invocation is admitted against kernel-resident state only: the registration of the requesting agent, the presence of the target tool in the registry, and the consistency of its semantic identity with the registered manifest.
The admission outcome does not depend on \texttt{mcpd}'s in-memory state at the moment of the decision.

\textbf{(2) No kernel-facing obligations for tool services.}
Tool authors write against an ordinary user-space IPC interface; the kernel-facing protocol is confined to \texttt{mcpd} and described in Section~\ref{sec:wire-protocols}.
This preserves compatibility with existing tool ecosystems and keeps incremental deployment practical.

\textbf{(3) Kernel-queryable observability.}
Admission, approval, and completion records are readable through kernel interfaces independently of \texttt{mcpd}'s liveness, so a daemon crash does not erase recent control-plane history.

What these commitments cover, and what they leave out (runtime semantic verification of tool behaviour, a general policy language, the broader attack surface of LLM-based systems), is fixed by the threat model of Chapter~\ref{chap:background}.



\subsection{System Architecture}
\label{sec:system-architecture}

\begin{figure}[t]
\centering
\includegraphics[width=0.9\linewidth]{figures/linux-mcp-overview.pdf}
\caption{Layered architecture of linux-mcp, showing the four runtime components and the wire protocols connecting them.}
\label{fig:linux-mcp-overview}
\end{figure}

The overall architecture of linux-mcp, shown in Figure~\ref{fig:linux-mcp-overview}, comprises four runtime components.

\texttt{llm-app} is the user-facing client.
It queries \texttt{mcpd} for the tool catalog, constructs invocation payloads from tool schemas, and submits requests.
The client has no direct interaction with the kernel module; all control-plane operations are mediated through \texttt{mcpd}.

\texttt{mcpd} is the user-space gateway and the semantic front-end of the control plane.
It is responsible for loading and validating tool manifests, computing manifest-derived semantic hashes, binding each session to the peer's \texttt{SO\_\allowbreak PEERCRED}-derived identity and registering that binding with the kernel, exposing catalog APIs (\texttt{list\_apps}, \texttt{list\_\allowbreak tools}), communicating with the kernel over Generic Netlink, and forwarding admitted requests to tool services over Unix domain sockets.
\texttt{mcpd} is the only component that sees both tool semantics and runtime endpoints, so it is the only one that needs changing when manifests or endpoints are reconfigured.

\texttt{kernel\_\allowbreak mcp} is the durable back-end of the control plane: a loadable kernel module implementing the \texttt{KERNEL\_\allowbreak MCP} Generic Netlink family.
It maintains the tool and agent registries, arbitrates invocation requests, manages approval ticket lifecycles, verifies the session binding hash on every request, pins a trust-on-first-use \texttt{binary\_\allowbreak hash} for each backend process, and records per-agent invocation history in a fixed-size \texttt{call\_\allowbreak log} ring buffer.
All kernel-held state is exposed read-only through sysfs under \texttt{/\allowbreak sys/\allowbreak kernel/\allowbreak mcp/}, which remains readable even when \texttt{mcpd} is not running.
The module does not parse manifests, execute tools, or implement a policy language; its only job is enforcing the admission-control invariants.

\texttt{tool-app} is the execution side: a collection of resident user-space services, each bound to a Unix domain socket.
When a request is admitted, \texttt{mcpd} forwards the payload to the appropriate tool service, which executes in an ordinary user-space environment and returns its result to \texttt{mcpd}, which then reports completion back to the kernel.
The directory \texttt{tool-app/\allowbreak manifests/} is the authoritative source of tool identity: \texttt{mcpd} reads manifests from here, hashes them, and registers the resulting digests with the kernel.

\subsection{Control-Plane Workflow}

\begin{figure}[t]
\centering
\includegraphics[width=0.85\linewidth]{figures/linux-mcp-workload.pdf}
\caption{Invocation lifecycle in linux-mcp.}
\label{fig:chapter3-lifecycle}
\end{figure}

The end-to-end path of a single invocation is:
\[
\texttt{llm-app} \to \texttt{mcpd} \to \texttt{kernel\_\allowbreak mcp} \to
\texttt{mcpd} \to \texttt{tool-app} \to \texttt{mcpd} \to \texttt{llm-app}
\]
Figure~\ref{fig:chapter3-lifecycle} illustrates this path as a sequence diagram.
This path captures one \texttt{tool:exec} invocation; a one-time \texttt{open\_\allowbreak session} handshake precedes the first such invocation on a given session and is described below.
A request originates in \texttt{llm-app}, which first issues a catalog query that \texttt{mcpd} satisfies from its manifest-derived in-memory state.
The kernel occupies this path not as a passive observer but as the active arbitration point: \texttt{mcpd} cannot forward a request to a tool without first obtaining an \texttt{ALLOW} decision from the kernel, and it cannot close an invocation without reporting \texttt{TOOL\_\allowbreak COMPLETE} back through the same channel.
Admission and completion are explicit, kernel-visible state transitions; the daemon no longer makes these decisions in private.

Before issuing any \texttt{tool:exec}, the client opens a session with an \texttt{open\_\allowbreak session} call.
On receiving this, \texttt{mcpd} reads the peer's credentials from the UDS connection via \texttt{SO\_\allowbreak PEERCRED}, hashes $(\text{uid}, \text{gid}, \text{pid}, \text{session\_id})$ into a binding hash, and registers the binding with the kernel.
This binding is the anchor the kernel re-verifies on every subsequent request from the session.

The client uses the catalog response to construct a structured invocation payload and submits it as a \texttt{tool:exec} call to \texttt{mcpd}; the complete message schema is given in \hyperref[sec:appendix-protocols]{Appendix~A.1}.

Upon receiving the request, \texttt{mcpd} performs a series of user-space checks: it resolves the active session, validates the payload against the declared input schema, identifies the target tool, and retrieves the semantic hash associated with that tool from its manifest-derived state.
Only upon successful completion of these checks does \texttt{mcpd} issue a \texttt{TOOL\_\allowbreak REQUEST} to the kernel, supplying the agent identifier, tool identifier, semantic hash, the session binding hash (derived from \texttt{SO\_\allowbreak PEERCRED} at \texttt{open\_\allowbreak session}), and the live-probed \texttt{binary\_\allowbreak hash} of the backend process as parameters.

The kernel module then performs arbitration.
It verifies that the requesting agent appears in the agent registry, that the target tool appears in the tool registry, that the semantic hash in the request matches the hash recorded at registration time, that the session binding hash in the request matches the one registered at \texttt{open\_\allowbreak session}, and that the \texttt{binary\_\allowbreak hash} (TOFU-pinned on the first non-empty value) still matches on this invocation.
Based on these checks and the tool's declared policy, the kernel returns one of three decisions. \texttt{ALLOW} lets the request proceed immediately and issues an approval ticket that the subsequent \texttt{TOOL\_\allowbreak COMPLETE} will reference. \texttt{DEFER} holds the request pending an explicit \texttt{APPROVAL\_\allowbreak DECIDE}, while \texttt{DENY} rejects it outright.
The detailed semantics of these decisions are described in Figure~\ref{fig:chapter3-approval}.

Depending on the kernel's decision, \texttt{mcpd} either forwards the request to the tool service, transitions the request into a pending state awaiting explicit approval, or terminates the invocation with an error.
After the tool service returns (with a normal result, an error, a timeout, or a crash caught by \texttt{mcpd}), \texttt{mcpd} issues \texttt{TOOL\_\allowbreak COMPLETE} to the kernel, referencing the ticket issued at admission; the kernel records the outcome in the per-agent \texttt{call\_\allowbreak log} and closes the invocation lifecycle from the control plane's perspective.

This workflow has an observability consequence: the kernel both opens and closes every invocation.
An invocation that never reaches \texttt{TOOL\_\allowbreak COMPLETE}, whether due to daemon failure or tool timeout, persists as an open entry in kernel state and remains inspectable via sysfs.
This contrasts with a purely user-space design, in which the lifecycle of a request is only as durable as the daemon's own in-memory representation.



\subsection{Wire Protocols}
\label{sec:wire-protocols}

User-space communication in linux-mcp runs over Unix domain sockets, while the kernel control-plane path uses Generic Netlink. The default UDS variant is filesystem-path (\texttt{uds\_rpc}). An abstract-namespace variant (\texttt{uds\_abstract}) is also implemented and exercised end-to-end by one demo manifest, and a VSOCK variant (\texttt{vsock\_rpc}) is reserved but not yet wired up. All UDS paths between \texttt{llm-app}, \texttt{mcpd}, and each tool service share the same framing of a 4-byte big-endian length prefix followed by a UTF-8-encoded JSON object, so a single framing routine serves both user-space endpoints. The Netlink path uses TLV-encoded attributes rather than JSON and is described at the end of this subsection.

\paragraph{llm-app to mcpd.}
\texttt{llm-app} sends two categories of message to \texttt{mcpd}: system control messages keyed by \texttt{"sys"} (covering \texttt{list\_\allowbreak apps}, \texttt{list\_\allowbreak tools}, \texttt{open\_\allowbreak session}, and \texttt{approval\_\allowbreak reply}) and tool execution requests keyed by \texttt{"kind": "tool:exec"}.
The two categories are distinguished by the top-level JSON key alone; \texttt{tool:exec} dominates in typical workloads, with \texttt{sys} messages appearing only at session boundaries.
On deferral, the \texttt{tool:exec} response carries a \texttt{ticket\_\allowbreak id} that the client submits with a subsequent \texttt{approval\_\allowbreak reply} to unblock the request.
Complete JSON schemas for all five messages are given in \hyperref[sec:appendix-protocols]{Appendix~A.1}.

\paragraph{mcpd to tool-app.}
Once the kernel returns \texttt{ALLOW}, \texttt{mcpd} forwards the request to the relevant tool service over a per-application UDS connection using a five-field request / five-field response schema.
Consistent with the design goal in Section~\ref{sec:design-goals} that tool authors should not need to reason about the control plane, the tool service sees only a \texttt{(req\_id, agent\_id, tool\_id, operation, payload)} request and returns \texttt{(req\_id, status, result, error, t\_ms)}; it is unaware of sessions, kernel tickets, or the Netlink protocol.
The \texttt{operation} field is the dispatch string a tool service routes on and is declared per-tool in the manifest; it is distinct from the \texttt{tool\_id} (numeric, used by the kernel for registry indexing) and from the tool's human-readable \texttt{name}.
The full field specification is given in \hyperref[sec:appendix-protocols]{Appendix~A.1} (Table~\ref{tab:mcpd-toolapp}).

\paragraph{mcpd to kernel\_mcp (Generic Netlink).}
The kernel control-plane interface uses the \texttt{KERNEL\_\allowbreak MCP} Generic Netlink family.
Messages carry TLV-encoded attributes, and each message type is registered as a Netlink \emph{command}; all commands except the read-only \texttt{LIST\_\allowbreak TOOLS} dump require \texttt{CAP\_\allowbreak NET\_\allowbreak ADMIN}.
The command set covering registration, invocation, approval, and registry observability is listed in Table~\ref{tab:netlink-cmds}.
The attributes carrying the invariants the kernel actually checks on the critical \texttt{TOOL\_\allowbreak REQUEST}/\texttt{TOOL\_\allowbreak DECISION} path are shown in Table~\ref{tab:netlink-attrs}; the full attribute set and a reference Python framing routine are given in \hyperref[sec:appendix-protocols]{Appendix~A.1} (Table~\ref{tab:netlink-attrs-full}).

The \texttt{BINARY\_\allowbreak HASH} attribute warrants specific mention because its production is hybrid.
\texttt{mcpd} computes the digest in user space: it opens a socket to the backend, resolves the peer PID via \texttt{SO\_\allowbreak PEERCRED}, and hashes the bytes of \texttt{/\allowbreak proc/\allowbreak <pid>/\allowbreak exe} at probe time.
For interpreter-hosted tools whose manifest declares an entrypoint script, the digest is a composite of the interpreter executable and the on-disk script bytes, re-read on every probe, so substituting the script between invocations invalidates the pin.
The kernel applies trust-on-first-use enforcement to whatever \texttt{BINARY\_\allowbreak HASH} arrives: the first non-empty value (whether on \texttt{TOOL\_\allowbreak REGISTER} or a subsequent \texttt{TOOL\_\allowbreak REQUEST}) locks the slot, moving its \texttt{binary\_\allowbreak hash\_\allowbreak state} from \texttt{unpinned} to \texttt{live\_\allowbreak pinned}; any later \texttt{TOOL\_\allowbreak REQUEST} whose \texttt{BINARY\_\allowbreak HASH} fails to match the pinned value is denied with reason \texttt{BINARY\_\allowbreak MISMATCH}.
The kernel never reads \texttt{/proc} itself; its role is to store and compare, not to probe.

\paragraph{Tool registration via manifests.}
Tool services do not self-register; each application is described by a JSON manifest in \texttt{tool-app/\allowbreak manifests/} from which \texttt{mcpd} derives semantic hashes and issues \texttt{TOOL\_\allowbreak REGISTER} Netlink commands at startup.
The separation of the semantic hash (what the tool is) from the binding fingerprint over \texttt{endpoint} and \texttt{transport} (where the backend is reached), and the rebind-on-address-change mechanism it drives, are detailed in Section~\ref{sec:manifest-semantics}.
The complete field specification and a representative manifest are given in \hyperref[sec:appendix-protocols]{Appendix~A.1} (Table~\ref{tab:manifest-fields}).


\subsection{Manifest Semantics and Tool Identity}
\label{sec:manifest-semantics}

Defining tool identity at the right granularity is a balance between two failure modes. Binding identity too tightly to transport details such as socket paths or process identifiers means that routine operational changes (redeployments, restarts, reconfiguration) invalidate registry state even when the tool itself has not changed. Defining it too coarsely makes the system unable to distinguish a semantically modified tool from one that has merely been redeployed at a different address.

linux-mcp resolves this tension by separating semantic identity from transport location.
Each tool is described by a JSON manifest, and the fields that fix its identity are tool identifier, tool name, application identifier, application name, risk tags, description, input schema, examples, path semantics, approval policy, and operation. These describe what the tool is intended to do, not where it runs.
The \texttt{operation} field is the RPC method actually invoked on the backend, so a silent edit retargets execution to a different code path and must change the tool's identity.
\texttt{mcpd} canonicalises these fields into a stable JSON serialisation and computes its SHA-256 digest, represented as a 64-character hexadecimal string.

The \texttt{endpoint} and \texttt{transport} fields are excluded from the semantic hash by design.
Both describe \emph{where} a backend is reached, not \emph{what} it does, and routine operational moves (switching a service to a new socket path, or between transport variants such as \texttt{uds\_rpc} and \texttt{uds\_abstract}) should not cause a tool to look semantically different to planners or to kernel-side hash checks.
\texttt{mcpd} therefore tracks these two fields in a separate \emph{binding fingerprint}, also a SHA-256 digest, maintained alongside but distinct from the semantic hash.
A change in either field produces a new binding fingerprint and drives an unregister-then-re-register cycle against the kernel. The cycle clears the kernel's trust-on-first-use \texttt{binary\_\allowbreak hash} slot on the grounds that the backend reached at the new address may not be the same process.
The semantic hash is unaffected by the move, so planners and approval records that reference the tool remain valid.

Any modification to a semantically significant field, including the input schema, risk tags, description, approval policy, path semantics, or \texttt{operation}, produces a different semantic digest, which the kernel detects as a mismatch on the next \texttt{TOOL\_\allowbreak REQUEST}.
The semantic hash is the single artefact both sides check, computed in user space from the manifest and verified by the kernel.



\subsection{Session Management and Approval Binding}

Approval would be straightforward if defer and execute were atomic, but they are not: the request that returns after the user decision has to prove it is the same request, in the same session, against the same tool, or a TOCTOU window~\cite{bishop2003computer, lee2021exprace} opens between the two events.
Each session is bound to the real peer credentials of its Unix domain socket connection~\cite{stevens2013unix}.
\texttt{mcpd} hashes \texttt{(uid, gid, pid, session\_id)}, where \texttt{session\_id} is per-session random, through BLAKE2b to an 8-byte output and stores the 64-bit digest as the binding hash alongside a session epoch.
Including \texttt{pid} and a random \texttt{session\_\allowbreak id} stops two processes sharing a UID from reusing each other's session.
The width matches the \texttt{AGENT\_\allowbreak BINDING} Netlink attribute, so \texttt{mcpd} resubmits the hash on every \texttt{TOOL\_\allowbreak REQUEST} and the kernel compares it against what was registered at \texttt{open\_\allowbreak session}.
A \texttt{DEFER} response creates an approval ticket tagged with this binding context and a five-minute expiry (overridable per request), and the kernel re-checks the binding when \texttt{APPROVAL\_\allowbreak DECIDE} arrives.
Session-scoped binding plus per-request re-verification rules out replay across sessions: the state at deferral is the state at execution.

\paragraph{Catalog-epoch rebind on manifest change.}
A second form of staleness arises when the catalog changes while a session is open: without further mechanism, a client could keep invoking a re-registered tool under its pre-change binding.
Each tool records a \texttt{registered\_\allowbreak at\_\allowbreak epoch} at (re-)registration, each session records an \texttt{opened\_\allowbreak at\_\allowbreak epoch} at open time, and a \texttt{TOOL\_\allowbreak REQUEST} is denied with \texttt{catalog\_\allowbreak stale\_\allowbreak rebind\_\allowbreak required} when the targeted tool's \texttt{registered\_\allowbreak at\_\allowbreak epoch} exceeds the session's \texttt{opened\_\allowbreak at\_\allowbreak epoch}.
Because the check is per tool, only the affected tool's re-registration invalidates a pending session; editing an unrelated manifest advances the shared \texttt{tool\_\allowbreak catalog\_\allowbreak epoch} counter without disturbing anything else.

Beyond the kernel ticket flow, linux-mcp supports two independent approval mechanisms: one in the kernel and one in the client, driven by different manifest fields and enforced at different layers.

\textbf{(1) Kernel-mediated approval (DEFER ticket flow).}
The kernel decides between \texttt{ALLOW}, \texttt{DENY}, and \texttt{DEFER} on every \texttt{TOOL\_\allowbreak REQUEST} based on the tool's \texttt{risk\_\allowbreak flags}, a bitmask derived from the manifest's \texttt{risk\_\allowbreak tags} at registration time.
Tools tagged \texttt{FILESYSTEM\_\allowbreak DELETE}, \texttt{DEVICE\_\allowbreak CONTROL}, \texttt{EXTERNAL\_\allowbreak NETWORK}, \texttt{PRIVILEGED}, or \texttt{IRREVERSIBLE} are deferred by default.
The five validity checks that distinguish \texttt{ALLOW} from \texttt{DENY}, and the state machine that governs the resulting tickets, are given in full in the Kernel Module Design section.
When a request is deferred, \texttt{mcpd} records the pending operation and returns the ticket identifier to the client.
After the user approves, \texttt{mcpd} first sends \texttt{APPROVAL\_\allowbreak DECIDE} to the kernel to transition the ticket, then replays the original request with the ticket identifier.
The kernel consumes the ticket during the second \texttt{TOOL\_\allowbreak REQUEST} and returns \texttt{ALLOW}.

\textbf{(2) Client-side local approval (payload-aware prompt).}
\texttt{llm-app} consults each tool's \texttt{approval\_\allowbreak policy} field before sending \texttt{tool:exec} to \texttt{mcpd}.
Most tools omit the field and incur no local prompt.
A tool may declare \texttt{approval\_\allowbreak policy.user\_\allowbreak confirmation} with a condition such as \texttt{when\,=\,"path\_\allowbreak outside\_\allowbreak repo"}, in which case the client resolves the payload's declared path fields against the repository root and prompts the user if any of them resolve outside.
The demo tool \texttt{write\_\allowbreak host\_\allowbreak text\_\allowbreak file} in \texttt{04\_\allowbreak desktop\_\allowbreak app.json} uses exactly this to require confirmation before writing to the host filesystem outside the project directory.
A declined local prompt terminates the request in user space.
No \texttt{tool:exec} is sent to \texttt{mcpd} and no kernel ticket is created.

\textbf{Relationship between the two mechanisms.}
The two mechanisms gate on different things, not the same checks.
Kernel \texttt{DEFER} is \emph{payload-agnostic}: every invocation of a high-risk-flagged tool is gated regardless of arguments.
The local prompt is \emph{payload-aware}: only requests whose arguments meet the declared condition are gated.
A single tool can trigger both, in which case execution proceeds only after the local prompt is approved, an \texttt{APPROVAL\_\allowbreak DECIDE} succeeds, and the ticket-bearing second \texttt{TOOL\_\allowbreak REQUEST} is admitted.

Figure~\ref{fig:chapter3-approval} illustrates the kernel-side state machine.
Approval always resolves \emph{before} tool execution, so no deferred request bypasses the gate and every deferred execution consumes an explicit ticket.
The kernel verifies only what it needs to: binding context preserved, ticket valid and unconsumed, and the \texttt{DEFER} $\rightarrow$ \texttt{ALLOW} transition explicitly authorised~\cite{klein2010sel4}.

\subsection{Kernel Module Design}

The kernel module \texttt{kernel\_\allowbreak mcp} is loadable and exposes the \texttt{KERNEL\_\allowbreak MCP} Generic Netlink family, with eight commands that cover registration, invocation, approval, and registry observability (\texttt{TOOL\_\allowbreak REGISTER}, \texttt{AGENT\_\allowbreak REGISTER}, \texttt{TOOL\_\allowbreak REQUEST}, \texttt{TOOL\_\allowbreak DECISION}, \texttt{TOOL\_\allowbreak COMPLETE}, \texttt{APPROVAL\_\allowbreak DECIDE}, \texttt{RESET\_\allowbreak TOOLS}, and \texttt{LIST\_\allowbreak TOOLS}).
Generic Netlink was chosen over \texttt{ioctl} or a character device because it provides structured TLV attributes with kernel-side schema validation, native support for dump-style enumeration (used by \texttt{LIST\_\allowbreak TOOLS}), and per-command \texttt{GENL\_\allowbreak ADMIN\_\allowbreak PERM} gating that integrates cleanly with Linux capabilities~\cite{netlink2026}.

The module's tool registry maps tool identifiers to their manifest-derived semantic hashes and associated metadata, with each entry also carrying risk flags, a per-tool \texttt{registered\_\allowbreak at\_\allowbreak epoch}, and the TOFU \texttt{binary\_\allowbreak hash} slot. A separate agent registry records agents permitted to submit requests through the control plane, together with the \texttt{binding\_\allowbreak hash}, \texttt{binding\_\allowbreak epoch}, and \texttt{opened\_\allowbreak at\_\allowbreak epoch} captured at \texttt{open\_\allowbreak session}.
Approval tickets are heap-allocated kernel objects with an explicit state machine.
The normal path runs \texttt{PENDING} $\to$ \texttt{APPROVED} $\to$ \texttt{CONSUMED}.
A reject takes \texttt{PENDING} to \texttt{DENIED}.
A post-approval revoke takes \texttt{APPROVED} to \texttt{DENIED}.
\texttt{EXPIRED} is implicit; an expired ticket is removed from the table, not marked with a stored state.

The kernel admits a \texttt{TOOL\_\allowbreak REQUEST} only when five conditions hold simultaneously, which keeps the arbitration logic narrow.
The submitting agent must appear in the agent registry, the target tool must appear in the tool registry, the semantic hash supplied by \texttt{mcpd} must match the hash recorded at registration, and the session \texttt{binding\_\allowbreak hash} must match the one captured at \texttt{open\_\allowbreak session}. If the TOFU slot is already pinned, the request's \texttt{binary\_\allowbreak hash} must match the pinned value as well.
Any failing check produces \texttt{DENY} with a specific reason code. Otherwise the kernel consults the tool's \texttt{risk\_\allowbreak flags}, and a tool carrying \texttt{FILESYSTEM\_\allowbreak DELETE}, \texttt{DEVICE\_\allowbreak CONTROL}, \texttt{EXTERNAL\_\allowbreak NETWORK}, \texttt{PRIVILEGED}, or \texttt{IRREVERSIBLE} is deferred and receives a fresh approval ticket, while everything else gets \texttt{ALLOW}.

\paragraph{Trust-on-first-use backend binary pinning.}
Semantic-hash admission guards what a tool claims to be, not which binary actually serves the socket.
The kernel therefore keeps a per-tool \texttt{binary\_\allowbreak hash} slot that \texttt{mcpd} fills on a live probe.
\texttt{mcpd} reads the backend's peer PID via \texttt{SO\_\allowbreak PEERCRED} and hashes \texttt{/\allowbreak proc/\allowbreak <pid>/\allowbreak exe}.
For interpreter-hosted tools the digest combines the interpreter with the on-disk script, so a script swap is still caught.
The slot locks on the first non-empty digest, which usually arrives on \texttt{TOOL\_\allowbreak REGISTER} but may arrive on the first successful \texttt{TOOL\_\allowbreak REQUEST} if the backend comes up later.
Any subsequent \texttt{TOOL\_\allowbreak REQUEST} whose digest does not match is rejected with reason \texttt{BINARY\_\allowbreak MISMATCH}.
A \texttt{binary\_\allowbreak hash\_\allowbreak state} field distinguishes \texttt{unpinned} from \texttt{live\_\allowbreak pinned}, since both previously read as an empty string.

\paragraph{Kernel-resident call log.}
Each agent carries a fixed-size 32-slot circular buffer of invocation records.
A record is a flat struct holding a monotonic sequence number, a nanosecond timestamp, the request and tool ids, a status (one of \texttt{OK}, \texttt{ERR}, \texttt{DENY}, \texttt{DEFER}), execution latency, a tool status code similar to a process exit code, 8-byte payload and response hash prefixes, and up to 48 leading bytes of any error output.
Completed invocations log from \texttt{cmd\_\allowbreak tool\_\allowbreak complete}; denied or deferred ones log from \texttt{cmd\_\allowbreak tool\_\allowbreak request}, so nothing escapes the audit buffer.
The kernel parses no JSON.
\texttt{mcpd} precomputes the hashes and passes them as fixed-width attributes.
The buffer is exposed through sysfs as a binary attribute and stays readable after a \texttt{mcpd} crash.
This provides crash-visible audit, not tamper-evident provenance: the buffer is bounded and the hashes are 8-byte prefixes.

\paragraph{Sysfs surface.}
Each tool directory under \texttt{/\allowbreak sys/\allowbreak kernel/\allowbreak mcp/\allowbreak tools/} exposes \texttt{name}, semantic \texttt{hash}, \texttt{binary\_\allowbreak hash}, \texttt{binary\_\allowbreak hash\_\allowbreak state}, and \texttt{registered\_\allowbreak at\_\allowbreak epoch}.
The global \texttt{tool\_\allowbreak catalog\_\allowbreak epoch} sits at the top of the tree.
Per-agent entries under \texttt{/\allowbreak sys/\allowbreak kernel/\allowbreak mcp/\allowbreak agents/} expose per-state counters (\texttt{allow\_\allowbreak count}, \texttt{deny\_\allowbreak count}, \texttt{defer\_\allowbreak count}, \texttt{completed\_\allowbreak ok\_\allowbreak count}, \texttt{completed\_\allowbreak err\_\allowbreak count}, \texttt{last\_\allowbreak exec\_\allowbreak ms}), a human-readable \texttt{last\_\allowbreak reason}, and the \texttt{call\_\allowbreak log} above with its head and count indices.
An operator can read what the kernel regards as authoritative without relying on daemon-side logs.



\subsection{Daemon Implementation}
\label{sec:daemon-implementation}

Most runtime work happens in \texttt{mcpd}; the kernel module is consulted only on admission decisions.
The daemon splits across four files: \texttt{manifest\_\allowbreak loader.py} (manifest parsing and semantic hashing), \texttt{session\_\allowbreak store.py} (session bindings), \texttt{netlink\_\allowbreak client.py} (kernel Netlink traffic), and \texttt{server.py} (JSON-RPC dispatcher and backend probe).

\paragraph{Kernel state reconciliation.}
The manifest directory is the source of truth, but the kernel registry survives across \texttt{mcpd} restarts.
The reload path has to keep the two in sync without invalidating existing sessions unnecessarily.
On each reload \texttt{mcpd} diffs the previous and new registries per tool and picks one of five reconcile actions.
\texttt{KEEP} applies when both fingerprints match, so no kernel call is issued.
\texttt{ADD} and \texttt{REMOVE} handle tools that appear or disappear.
\texttt{RE\_\allowbreak REGISTER\_\allowbreak SEMANTIC} covers a changed semantic hash on an unchanged backend; the in-place re-register preserves the kernel's TOFU \texttt{binary\_\allowbreak hash} slot.
\texttt{RE\_\allowbreak REGISTER\_\allowbreak BINDING\_\allowbreak MOVE} fires when \texttt{transport} or \texttt{endpoint} changed: the backend may now be a different process, so \texttt{mcpd} unregisters first to clear the TOFU slot, then re-registers so the next probe pins a fresh digest.

\paragraph{Probe-failure fail-closed.}
The TOFU \texttt{binary\_\allowbreak hash} mechanism only works if \texttt{mcpd} never quietly substitutes an empty digest for a failed probe.
When \texttt{mcpd} cannot reach the backend, cannot read \texttt{/\allowbreak proc/\allowbreak <pid>/\allowbreak exe}, or does not recognise the transport, the probe returns a non-live result and the exec path refuses the invocation.
No empty \texttt{BINARY\_\allowbreak HASH} flows upward. A failed probe will not reset a pinned slot and let a swapped backend start clean, and a cold registration before the backend comes up will not install a blank pin a later swap could inherit.
Audit records for probe refusals carry \texttt{source=probe}, distinct from kernel denials (\texttt{source=arb}) and client declines (\texttt{source=user\_\allowbreak decline}).

\paragraph{Netlink client.}
\texttt{mcpd} keeps one long-lived Netlink socket to \texttt{kernel\_\allowbreak mcp} rather than reopening one per request.
Socket setup is measurable on a microsecond-scale path, and a persistent connection simplifies ordering \texttt{APPROVAL\_\allowbreak DECIDE} against its eventual \texttt{TOOL\_\allowbreak REQUEST} replay.
Optional per-stage latency counters (session lookup, kernel arbitration, tool execution, end-to-end) plug into this path for the overhead analysis in Chapter~\ref{chap:evaluation}.

\paragraph{Deployment and fail-closed configuration.}
\texttt{mcpd} needs \texttt{CAP\_\allowbreak NET\_\allowbreak ADMIN} for Netlink administration and \texttt{CAP\_\allowbreak SYS\_\allowbreak PTRACE} for reading \texttt{/\allowbreak proc/\allowbreak <pid>/\allowbreak exe} across uid boundaries during the backend probe.
The supplied systemd unit runs \texttt{mcpd} as a dedicated unprivileged user, grants both caps via \texttt{Ambient\allowbreak Capabilities}, and bounds them in \texttt{Capability\allowbreak Bounding\allowbreak Set} so an \texttt{execve}'d child cannot widen the cap set.
Two further directives close gaps the threat model would otherwise leave open: \texttt{No\allowbreak New\allowbreak Privileges=true} blocks SUID-based privilege escalation in any child, and \texttt{Memory\allowbreak Deny\allowbreak Write\allowbreak Execute=true} rejects writable executable pages and so blunts a memory-corruption exploit class.
The remaining systemd hardening (\texttt{Protect\allowbreak System=strict} with a \texttt{/\allowbreak tmp} allowlist, \texttt{Protect\allowbreak Kernel\allowbreak Tunables}, \texttt{Protect\allowbreak Kernel\allowbreak Modules}, \texttt{Protect\allowbreak Control\allowbreak Groups}, \texttt{Restrict\allowbreak SUIDSGID}, \texttt{Lock\allowbreak Personality}, \texttt{Private\allowbreak Devices}) shrinks the blast radius of a compromised daemon at no operational cost.

A separate question is which UIDs may serve as tool backends.
\texttt{mcpd} checks \texttt{SO\_\allowbreak PEERCRED} against \texttt{allowed\_\allowbreak backend\_\allowbreak uids} on every probe and exec dial.
Defaulting to \texttt{\{0\}} (root only) rejects every realistic non-root backend and leaves \texttt{binary\_\allowbreak hash} unpinned; defaulting to \texttt{\$SUDO\_\allowbreak UID} folds the launcher script's correctness into the daemon's trust boundary.
\texttt{mcpd} therefore refuses to start without an explicit TOML list or an explicit \texttt{LINUX\_\allowbreak MCP\_\allowbreak TRUST\_\allowbreak SUDO\_\allowbreak UID} opt-in; backend UIDs come from configuration, never from the launch environment.



\subsection{State Reconciliation and Recovery}
\label{sec:reconciliation}

linux-mcp treats a daemon restart as a recovery problem, not a process-management detail.
The three layers of state it keeps sit on different sides of the crash boundary.
The kernel holds the tool registry, approval tickets, the \texttt{tool\_\allowbreak catalog\_\allowbreak epoch} counter, and each agent's \texttt{call\_\allowbreak log}.
All of that survives a \texttt{mcpd} crash.
The manifest set on disk is the authoritative source of tool identity and is unaffected by restart.
Open sessions, peer-credential bindings, and pending request maps live only in \texttt{mcpd}'s memory and are discarded when the process exits.

The cold-start procedure follows directly from this partition.
At startup, \texttt{mcpd} issues \texttt{RESET\_\allowbreak TOOLS} to clear the kernel tool registry and then re-registers each manifest tool unconditionally, which guarantees the kernel and the on-disk manifest set agree before any client is accepted.
\texttt{RESET\_\allowbreak TOOLS} alone does not bump the catalog epoch; the per-tool register calls that follow do.
A surviving session holding a pre-restart \texttt{opened\_\allowbreak at\_\allowbreak epoch} will therefore receive \texttt{catalog\_\allowbreak stale\_\allowbreak rebind\_\allowbreak required} on its next request against any re-registered tool.
Cold start deliberately does not attempt diff-based reconciliation against the surviving kernel registry: the per-tool diff path described in Section~\ref{sec:daemon-implementation} applies only to live manifest reloads while \texttt{mcpd} is running.
Optimising cold start to reuse residual kernel state under an explicit divergence signal is left to future work (Section~\ref{sec:further-work}).

This split preserves audit and admission state without pretending that a crashed daemon can safely continue pre-existing sessions.
Approval tickets and registry history stay inspectable through sysfs across the restart, so an operator can still read what recent activity occurred while \texttt{mcpd} was down or unresponsive.
Any surviving ticket whose referenced session no longer has a live \texttt{mcpd}-side binding simply expires at its TTL.
Clients reopen any session that pre-dates the restart on their first post-restart request.
The client library handles this rebind automatically when it sees the \texttt{catalog\_\allowbreak stale\_\allowbreak rebind\_\allowbreak required} reason.

A kernel module reload sits outside this recovery model.
Unloading \texttt{kernel\_\allowbreak mcp} drops all kernel-held state (registry, tickets, epochs, call logs), so the operator must treat the next start as a fresh system.
\texttt{mcpd} re-registers from manifests as usual, but any historical audit kept only in the now-gone kernel buffer is lost.



\subsection{Rationale for the Kernel--User-Space Split}
\label{sec:design-alternatives}

Section~\ref{sec:related-work} established the main layer mismatch: MCP admission decisions depend on parsed request state and manifest-derived tool identity, while eBPF/LSM hooks observe kernel objects and syscall events.
In implementation terms, this ruled out an eBPF-only design because it would have required reconstructing MCP framing and request identity from raw socket bytes under verifier constraints.
Direct LSM hooks had the same attachment-point problem: no hook naturally carries both the request body and the registered tool identity.

A capability-style design was also considered, but it would have changed the programming model for tool services.
Each backend would need to obtain, store, and present kernel-facing capabilities during invocation.
That conflicts with the design goal that tool authors only provide a JSON manifest and continue to speak ordinary user-space RPC.

Generic Netlink was chosen because it gives \texttt{mcpd} a structured kernel interface for exactly the state transitions this design needs: tool registration, agent registration, request arbitration, approval decisions, completion reporting, and registry dumps.
It keeps JSON parsing, schema validation, and tool execution in user space, while letting the kernel hold the small amount of durable state needed for admission.
The cost is a loadable kernel module rather than a policy hook or BPF program, but the module remains limited to registry membership, ticket state, session binding, and audit records.
